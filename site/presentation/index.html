bomi

Building and Validating a USB LED Flashlight BOM

Search parts, compare options, select BOM entries, validate stock and cost, and keep everything version-controlled.

Project Goal

What We Build

A compact USB-C powered LED flashlight board using a PT4115 driver and two brightness control switches.

Chapter 1

Project Setup

Initialize repository and create the project BOM baseline.

Create project workspace

Concept

  • Start with a clean project folder under demo workspace.
  • Use normal git workflow from the beginning.
  • Keep all BOM changes inspectable as diffs.
Key Commands
mkdir -p usb-led-flashlight

Create the demo project folder.

git init usb-led-flashlight

Start version tracking immediately.

Initialize bomi project metadata

Concept

  • .bomi/project.yaml becomes project BOM source-of-truth.
  • Metadata is lightweight and repository-friendly.
Key Commands
bomi init --name "usb-led-flashlight" --description "..."

Create `.bomi/project.yaml` for this project.

ls -la .bomi

Confirm project metadata files were created.

Chapter 2

Search and Inspect Parts

Use live search to shortlist components and inspect selected part metadata.

Search core parts

Concept

  • Search returns stock, package, price, and type in one step.
  • Keyword quality directly affects candidate quality.
Key Commands
bomi search <term> --limit n

Find components matching a term and cap results at `n`.

bomi search "PT4115 LED driver" --limit 5

Shortlist driver candidates with stock/price data.

Inspect a driver part in detail

Concept

  • fetch caches exact part details for repeatable offline inspection.
  • info surfaces key ratings and datasheet links.
Key Commands
bomi fetch <lcsc_code>

Cache exact part details locally.

bomi info <ref_or_lcsc>

Inspect ratings, attributes, and datasheet links.

Search supporting power parts

Concept

  • Use targeted searches for diode, inductor, and switch candidates.
  • Filter for practical inventory and package constraints.
Key Commands
bomi search "Schottky diode SOD-123" --basic-only --limit 5

Find low-cost, assembly-friendly diode options.

bomi search "inductor 47uH SMD" --limit 5

Find suitable buck inductor candidates.

Chapter 3

Compare and Decide

Evaluate trade-offs and document constraints before final selection.

Compare switch options

Concept

  • Compare makes trade-offs explicit in a single table.
  • Catalog metadata gaps are part of real engineering workflow.
Key Commands
bomi compare <lcsc_a> <lcsc_b>

Compare specs and trade-offs side by side.

bomi compare C49023761 C145837

Evaluate switch options quickly.

Chapter 4

Build and Validate BOM

Select final parts, review project status, run checks, and export data.

Select BOM parts

Concept

  • Each select binds a reference designator to an LCSC part.
  • Notes capture intent and aid future reviews.
Key Commands
bomi select <lcsc_code> --ref <ref> --qty n --notes "..."

Add or assign a part to a BOM reference.

bomi select C347356 --ref U1 --qty 1 --notes "PT4115 LED driver"

Record one concrete project selection.

Review BOM and project status

Concept

  • list shows resolved parts with cost and stock context.
  • status summarizes count, estimated cost, and warnings.
Key Commands
bomi list

Show the full resolved project BOM.

bomi status

Summarize cost, count, and warnings.

Refine, check, and export

Concept

  • Reference updates are first-class operations (relabel).
  • list --check refreshes BOM health with live data.
  • CSV/JSON exports support downstream scripts and reviews.
Key Commands
bomi relabel C1 C99; bomi relabel C2 C1; bomi relabel C99 C2

Swap references safely using a temporary designator.

bomi list --check

Refresh stock/price and validate BOM health.

Create artifacts and inspect git changes

Concept

  • Save datasheet PDFs for critical parts as review artifacts.
  • Use git status to confirm exactly what changed.
Key Commands
bomi datasheet <codes...> --pdf -o artifacts/datasheets

Capture datasheet PDFs as project artifacts.

git status --short

Review all changes before committing.

Recap

Workflow Summary

We moved from ad-hoc part selection to a repeatable workflow with traceable decisions.

We validated stock and pricing early, captured comparison trade-offs, exported BOM artifacts, and kept all part decisions in version control for review and reuse.
